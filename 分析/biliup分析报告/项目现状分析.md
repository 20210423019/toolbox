# BiliUp 项目现状分析

> 面向对象：项目维护者 / 后续开发者
> 分析时间：2026-07-07
> 分析范围：当前工作目录整体（`biliup/` 源码 + `biliup分析报告/` 文档 + 根目录两个 exe）

---

## 1. 项目定位

**BiliUp** 是一个基于 **Tauri 2 + React** 的 B 站投稿管理桌面客户端。它的本质是把开源命令行工具 **biliup-cli**（`biliup.exe`）封装成图形界面，并自行实现扫码登录来产出 Cookie，再交给 CLI 去执行上传、下载等 8 个功能。

这正好对应 `biliup分析报告/` 里已有的可行性结论——**采用"进程调用"集成方案**（用 `tauri-plugin-shell` 调用 `biliup.exe`），而非把 Rust 库集成进 Tauri。当前源码正是该方案的落地实现。

---

## 2. 技术栈

| 层 | 技术 |
|----|------|
| 前端 | React 19 + TypeScript + Vite 7 |
| 后端 | Rust + Tauri 2 |
| 进程集成 | `tauri-plugin-shell`（调用 `biliup.exe`） |
| 登录直连 | `reqwest` 直连 B 站 passport API（绕过浏览器 CORS） |
| 打包 | Tauri `externalBin` → `binaries/biliup-x86_64-pc-windows-msvc.exe` |
| 权限 | `shell:allow-execute` + `shell:allow-spawn`（较宽） |

## 4. 代码规模

| 语言 | 文件 | 行数 | 说明 |
|------|------|------|------|
| Rust | 5（`main`/`lib`/`commands`/`biliup_cmd`/`login_api`） | ~396 | 后端逻辑精炼，模块划分清晰 |

> 结论：代码量小、结构清晰，但前端**过度集中在单个 `App.tsx`**，缺乏组件拆分与状态管理（无路由、无全局 store），随功能增长会难以维护。

## 5. 架构图

```
┌─────────────────────────────────────────────────────────────┐
│  React 前端 (App.tsx)                                        │
│  · 扫码登录 UI   · 手动 Cookie 输入   · 命令调试面板          │
└───────────┬───────────────────────────────┬─────────────────┘
            │ invoke()                        │ invoke()
            ▼                                 ▼
┌──────────────────────┐          ┌────────────────────────────┐
│ login_api.rs (Rust)  │          │ commands.rs (Rust)         │
│ 扫码生成 + 轮询       │          │ 8 个 Tauri Command          │
└───────────┬──────────┘          └──────────────┬─────────────┘
            │ reqwest HTTPS                      │ biliup_cmd.rs
            ▼                                    │  .run() → shell
┌──────────────────────┐                        ▼
│ B站 passport API      │          ┌────────────────────────────┐
│ passport.bilibili.com │          │ biliup.exe (CLI 子进程)     │
└──────────────────────┘          │ 真正的上传/下载/列表逻辑     │
                                   └──────────────┬─────────────┘
                                                 ▼
                                   ┌────────────────────────────┐
                                   │ B站投稿/下载 API + cookies.json│
                                   └────────────────────────────┘
```

设计上的亮点是：**登录走 Rust 直连**（避开前端 CORS 与 Cookie 跨域问题），**其余功能走进程调用**（复用成熟 CLI，不重复造轮子）。思路正确。

## 6. 功能实现状态 ⚠️ 关键发现

| 模块 | 状态 | 说明 |
|------|------|------|
| 扫码登录 | ✅ 完整 | 生成二维码 → 轮询 → 提取 4 个 Cookie 字段 |
| 手动 Cookie 输入 | ✅ 完整 | SESSDATA + bili_jct，可复制 JSON |
| 8 个命令按钮 | ⚠️ **占位/演示态** | 见下方说明 |
| 真实上传/列表/追加表单 | ❌ 未实现 | 没有真正的参数输入 UI |

**重点**：前端 `App.tsx` 里的"可用命令"面板调用 `invokeCommand()` 时，传的是**硬编码假参数**：

- `show` → 固定 `vid: "BV1xx"`
- `download` → 固定 `url: "https://live.bilibili.com/12345"`
- `dump-flv` → 固定 `filePath: "test.flv"`
- `upload` / `append` / `list` → 传空 `opts: {}`

也就是说，目前项目**只是"登录器 + 命令调试面板"**，真正面向用户的投稿/下载工作流 UI 还没有做。后端 8 个命令本身是正常的，但前端没有把真实表单数据接进去。

## 7. 发现的问题与风险

| # | 位置 | 问题 | 严重程度 |
|---|------|------|----------|
| 1 | `login_api.rs` `extract_cookies_from_response` | `resp.headers().get("set-cookie")` **只返回第一个 Set-Cookie 头**；B 站把 SESSDATA / bili_jct / DedeUserID / ac_time_value 分多个响应头下发，会漏掉。当前主要依赖 `data.url` 里的查询参数回退才工作，逻辑脆弱 | 中 |
| 2 | `App.tsx` `invokeCommand` | 命令区用硬编码假参数，是演示桩而非真实功能 | 高（功能缺口） |
| 3 | `App.tsx` QR 生成 | 用第三方 `api.qrserver.com` 生成二维码图，会把含 `qrcode_key` 的登录 URL 发给外部服务 | 中（隐私） |
| 4 | `tauri.conf.json` | `"csp": null` 关闭了内容安全策略 | 低 |
| 5 | `biliup_cmd.rs` | `cookie_path` 用相对路径 `cookies.json`，依赖进程 CWD；打包后路径不确定 | 中 |
| 6 | `App.tsx` | Cookie 明文展示并复制到剪贴板（敏感凭据暴露） | 中 |
| 7 | `capabilities/default.json` | `shell:allow-execute` + `shell:allow-spawn` 权限较宽，可生成任意进程 | 中 |

**建议修复**：
- 问题 1：改用 `resp.headers().get_all("set-cookie")` 遍历所有头。
- 问题 2：为每个功能做真实表单，从输入框取参后 `invoke`。
- 问题 3：本地生成二维码（Rust `qrcode` crate 或前端 `qrcode` 库）。
- 问题 5：用 `app.path().app_data_dir()` 存放 `cookies.json`。
- 问题 7：将 shell 权限收敛为仅允许 `biliup` 二进制路径。

### 源码阅读新发现

1. **项目实际上是 3-crate workspace**（biliup / biliup-cli / stream-gears），而非单 crate。
2. **stream-gears**（Python 绑定）是既有分析完全未覆盖的模块，使用 PyO3 + Maturin 导出核心函数给 Python 调用。
3. **Web 管理服务**（`server` feature, Axum + SQLite + Actor 模式）也是分析盲区，提供完整的 REST API 和自动化录制-上传管线。
4. **双客户端设计**：`StatelessClient`（无状态，用于下载）和 `StatefulClient`（带 CookieStore + buvid，用于登录上传）。
5. **AppKey 多平台签名**：BiliTV 和 Android 两套 app_key/appsec，对应不同端 B站 API。
6. **buvid 生成**：模拟随机 MD5 字符串作为设备指纹，适配 B站反爬机制。
7. **API 签名**：所有请求使用 `MD5(params + appsec)` 签名，`Credential::sign()` 统一实现。
8. **二维码轮询无超时**：`login_by_qrcode` 中 loop 没有 timeout 保护。
9. **部分代码已注释**：kodo/cos 上传线路已注释，仅 Upos 实际激活；client.rs 接口 submit/edit 标记为 `#[deprecated]` 自动回退到 app/web。
10. **密码登录的 unwrap 风险**：`RsaPublicKey::from_public_key_pem(&pub_key).unwrap()` 可能因 API 变更 panic。

## 11. ToolBox.exe 安全警示 ⚠️

根目录的 `ToolBox.exe`（约 12.7 MB）是一个**独立于 biliup 项目之外**的可执行文件（不在源码树内，与本项目功能无关）。对其做的静态特征扫描结果异常：

- PE 头有效（`MZ`），确为 Windows 可执行文件；
- 但 `strings` 扫描**找不到任何长度 ≥ 6 的可读 ASCII 字符串**（正常程序一定有导入表 DLL 名、区段名等）；
- 无任何运行时标记（`.NET` / Rust / Go / `UPX` 等特征均不存在）。

> 一个正常的 PE 文件几乎不可能"0 条可读字符串"。这强烈暗示该文件被**加壳、加密或做了反字符串提取混淆**。无论其真实用途如何，**在明确来源之前不应直接运行**。

**建议**：先确认 `ToolBox.exe` 的确切来源与用途（谁放的、做什么用），再做处理。如无法确定，建议隔离并杀毒扫描。
